# Feature Specification: Portability and Deletion

**Feature ID**: `010-portability-and-deletion`
**Status**: Readiness/analyze PASS — implementation authorized
**Canonical bible**: [Hekayati product specification](../001-hekayati-product-bible/spec.md)

This document is an ownership and acceptance slice. Canonical requirement wording and shared rules remain in the bible. On conflict: constitution → bible → this slice.

## Outcome

The operator can export a complete, versioned, secret-free project snapshot; safely import hostile or older archives through explicit modes; and permanently delete selected customer/project data with a verified dependent-media report.

## Requirements _(mandatory)_

Primary requirement ownership: **FR-005 and FR-125–129**.

Primary user journey: **US9**. Primary clarification: **C-07**. Full routing: [migration map](../MIGRATION.md).

Owned capability boundaries:

- Pre-deletion affected-item inventory, shared-character decisions, explicit confirmation, job cancellation, DB cascade, filesystem media removal, Studio history/assets, and verification report.
- Paused consistent export snapshot, versioned manifest, checksums, complete project contents, unrelated-data exclusion, and mandatory secret scan.
- Validated import modes, ID/customer conflict handling, older-schema migration, future-schema rejection, path/symlink/executable defenses, disk precheck, staging, and atomic commit/rollback.
- Honest export-is-not-backup messaging in the lifecycle workflow.

Customer/family/character ownership and dependency references come from feature 003. Job cancellation and snapshot safety use feature 006. Complete generated assets/approvals/PDFs come from features 007–009. Slice 010 proves the typed 011 registration seam with a synthetic participant; feature 011 owns the real standalone Studio history/assets participant and repeats combined evidence before its own checkpoint and Phase 10.

## Dependencies and interfaces

- Deletion of library-only data depends on feature 003; deletion of generated projects additionally depends on feature 007 and scheduler cancellation from 006.
- Full-fidelity export/import depends on the current schema and asset roles from every completed upstream feature, including Studio records when their owning customer is in scope.
- The feature may proceed after 007 in parallel with 008/009, but final round-trip coverage expands when their PDF artifacts exist.
- Secret scans consume redaction/credential patterns from features 002 and 005; no archive may be released when scanning fails.

## User Scenarios & Testing _(mandatory)_

Canonical story and scenarios: **US9** in the [product bible](../001-hekayati-product-bible/spec.md).

Independent acceptance: export a completed project, prove checksums and zero secrets/unrelated data, corrupt and weaponize fixture archives to confirm pre-write rejection, import a valid archive into a fresh instance with full fidelity, interrupt an import to prove rollback, then permanently delete and verify every reported DB/media dependency—including a synthetic customer-owned 011 participant fixture—is gone. Feature 011 repeats this evidence with its real schema.

## Success Criteria _(mandatory)_

Primary measurable outcome: **SC-005**. CHK024, CHK209, CHK217–CHK219, and every canonical US9 scenario provide the remaining round-trip, deletion, and hostile-archive evidence.

## Clarified delivery contract

1. The v1 destructive target union is exactly `customer | project`. Routine customer/family/character/look removal remains reversible archive/restore. Project deletion removes the project-owned graph while preserving customer/family/library/Studio data. Customer deletion removes every owned family, character, reference photo, project, generated output, customer-scoped Studio record, managed export, and dependent job/audit record. Global settings, seed/operator templates, and reusable printer profiles are not customer-owned and remain.
2. T-P9-01 owns a typed frozen `PortabilityParticipant` registry as the only ownership graph. Every participating collection declares its strict schema/version, export modes, owner selector, dependencies, asset/original references, explicit ID-rewrite function, derived-field/hash rebase function, import migrations, deletion order, and verification query. Generic recursive string replacement and ad-hoc collection scans are forbidden. Startup/tests fail on duplicate participants or any known project/customer-bearing collection, asset role, job lineage, or direct scoped SQL writer omitted from the completeness catalog. Slice 010 registers the real 003–009 graph plus a synthetic 011 contract fixture; slice 011 later owns the real Studio entry.
3. Export is a durable local operation. One transaction acquires the project scope in `draining`, sets the project pause, moves queued/blocked project jobs to operator pause, blocks new scoped mutations and scheduler admission, and records the exact already-claimed/running attempts. Only those captured attempts may finish while draining; the operator waits or explicitly cancels them. No snapshot starts while a mutating attempt can still commit, and no attempt may commit after the lock enters `snapshot`. Waiting-review and terminal history may be captured. Export never resumes work automatically.
4. A consistent export never keeps a SQLite transaction open while reading or writing files. After quiescence, one synchronous immediate transaction rechecks lock/project/job revisions, freezes the exact schema-validated canonical document bytes, inclusion reasons, IDs/schema versions, ordered asset/original ID+metadata+checksum inventory, idempotent media holds, registry hash and snapshot hash into durable bounded snapshot rows, then commits. Asynchronous staging reads only those rows and held managed files, verifies authoritative streamed byte counts/checksums, and never rereads live documents. Missing/drifted media fails with no candidate. After every byte is privately staged, one transaction releases holds and the project lock; packaging and both secret scans consume only staged bytes. No transaction spans an `await` or file stream.
5. Project export contains only the selected project graph and required same-customer references: the selected customer/family; exact referenced character/look/reference-photo versions; character sheets; template version; story/scenes/prompts/pages/reviews; layouts/cover/approval evidence; required scheduler lineage; generated assets; previews; print runs/reports/proofs/PDFs when present; and customer-scoped Studio history/assets when registered by feature 011. It excludes other customers, other projects, unrelated family members/versions, global settings, credentials, logs, incidents, temporary files, prior export archives, and unreferenced assets. Every included record is reachable from a manifest root or an explicitly named customer-scoped Studio root.
6. Exact uploaded originals may leave the private namespace only through this explicit export, as FR-021 permits. The export UI names the child-photo scope before start. Archives store original and derived/generated bytes under fixed generated entry names; no local source path, data-root path, operator-selected path, Keychain/Codex path, command line, runtime token, or temporary name appears in a manifest or entry.
7. Current archive format is the single strict pair `format: "HekayatiArchive"` and `manifestVersion: 2`, named `HekayatiArchive/v2`. `manifest.json` also contains app version, created-at, export ID, mode/scope/root IDs, deterministic document and media entries, each entry's byte count/SHA-256/content kind/schema version, aggregate uncompressed bytes, and a canonical snapshot hash. It lists every entry except itself; the finalized ZIP checksum is stored in the local export record. Entry order is deterministic. Missing, extra, duplicate, or checksum-unlisted entries reject. `ArchivePolicy/v1` is independently the resource/security policy version, not an archive format.
8. `HekayatiArchive/v1` is the one frozen supported older fixture, encoded exactly as `format: "HekayatiArchive"` plus legacy `schemaVersion: 1`, and has an explicit pure v1→v2 manifest/document migration. Migration occurs only in staging and never edits the source archive. Unknown older versions and every version greater than v2 reject before product writes with a bounded version error. Each contained domain document also passes its participant's ordered schema migrations; missing or future document migrations reject.
9. Export secret scanning is a release gate, not redaction. The shared exact-runtime-secret registry plus fixed Gemini, bearer/private-key, Codex-auth/auth-file, Keychain-dump, cookie/token and seeded-canary patterns stream-scan canonical documents, entry names, all staged bytes, and a second independent read of the finalized ZIP entries. Any match destroys the candidate, records only bounded pattern/category/path facts, creates no ready/downloadable archive, and leaves the prior valid export unchanged. SC-005 also scans the DB and complete log corpus.
10. Ready archives live only in the managed private export root with 0700/0600 permissions and an indexed checksum; APIs never accept or return a filesystem path. Download verifies ready state, scope, checksum, integrity and safe attachment headers at request time. The UI repeats that export is manual portability, not automatic backup, and that copies saved outside Hekayati cannot be found or deleted by Hekayati.
11. Import first streams the operator-selected external read-only ZIP into an opaque 0600 staging reservation under the managed data root. Hekayati never mutates or deletes the operator source and owns/cleans only its generated reservation. Central-directory validation is lazy and bounded. `ArchivePolicy/v1` fixes these non-archive-adjustable maxima: 8 GiB compressed upload, 20,000 entries, 240 UTF-8 bytes per entry name, 8 MiB manifest, 16 MiB canonical JSON document, 2 GiB single media/PDF entry, 16 GiB aggregate uncompressed bytes, and 200:1 per-entry and aggregate compression ratios. Before extracting any entry it rejects encryption, multi-disk archives, malformed/truncated ZIPs, absolute/drive/UNC/backslash/NUL/empty/dot/traversal paths, non-NFC or case-fold canonical-name collisions, duplicates, symlinks/non-regular entries, executable mode/content, nested archives, unlisted entries, or any policy-limit violation. Only the closed v2 path/MIME/content-kind allow-list may stage. Changing a maximum requires a versioned policy/spec change; neither an archive nor request may raise it.
12. Disk preflight runs before extraction and again before commit. It reserves the declared uncompressed bytes plus bytes for new content-store files plus the larger of twice the canonical document bytes or 256 MiB, while preserving the configured disk-warning reserve. False manifest sizes, integer overflow, or insufficient free space reject with zero extracted/product state. Streaming byte limits remain authoritative even after the estimate passes.
13. Import validates manifest/schema, every canonical JSON document, referential closure, checksums, byte counts, media magic/decode where applicable, PDF/ICC/template facts through their owning participants, asset metadata compatibility, secret absence, and mode eligibility entirely in staging. Validation produces one immutable `ImportPlan` with source snapshot hash, target mode, explicit customer/family/project mappings, a `create_from_archive | map_existing_same_customer` customer resolution, complete archive-ID→local-ID map, conflicts, preserved/dropped records, authorization/consent consequences, expected writes/releases, disk facts, and confirmation hash. Mapping an existing customer/family requires its exact revision plus an explicit same-real-customer attestation; a different-customer shortcut rejects. The current local customer/contact and active consent records are never silently overwritten, and current local consent remains authoritative for any later provider dispatch. A changed source, target revision, mapping, attestation, or plan hash requires a new plan.
14. ID rewriting is field-aware and total. `as_new_project` assigns fresh local IDs for project/customer/family/character/look/domain/job/history records and rewrites every declared reference, dependency, input snapshot and provenance reference. Participants then recompute every ID-derived request, idempotency, authorization, approval, provenance, layout, preview, print-source and run hash in dependency order; no pre-remap derived value is trusted, and strict schema/closure/hash validation runs again over the complete rebased graph. Approval evidence remains current only when its participant proves the approved semantic content is byte-for-byte unchanged by the administrative ID rebase and the customer resolution is valid; otherwise it is retained only as historical evidence and current content/print authorization is cleared for fresh approval. Exact compatible media already in the content-addressed stores may map to the existing canonical local asset/original ID with a refcount increment; a same ID with different bytes/metadata never aliases. Archive IDs are never trusted as local authority.
15. `replace_existing` requires an exact target project, owner/family mapping, target revision, affected-item plan and typed confirmation. It never overwrites shared customer/family/character/template/printer-profile records implicitly: each required shared reference is explicitly mapped or imported under a fresh ID. Confirmation force-cancels all nonterminal target-project work, replaces only the project-owned graph in one transaction, decrements old references, and leaves unrelated projects/Studio/library data byte-identical.
16. `characters_only` imports only selected customer/family/character/look/reference-photo records and their required media into a new or explicitly mapped family; it creates no project/story/page/job/approval/PDF/Studio record. `templates_only` imports only strict parameterized template identities/versions; legacy private/customer/character/photo/mention references are stripped into role slots by the 004 privacy extractor before commit or the import rejects. Selective modes cannot smuggle disallowed records through unlisted references.
17. Project archives may include project-scoped scheduler lineage required by assets, approvals and print evidence, but never global provider/quota/credential/storage incidents or runtime leases/tokens. Export quiescence guarantees no claimed/running job. On import, terminal history stays terminal; waiting human gates retain exact remapped targets; every other nonterminal executable job is normalized to `paused(operator)` with no lease/retry timer. Job request hashes and idempotency keys are recomputed from the rebased canonical request rather than copied. The imported project remains paused until an explicit normal resume and current capability/consent/reference checks run again.
18. Commit uses an operation-owned hierarchical target lock, prepared content-addressed files, bounded hashed ledger pages, and one SQLite transaction. Fully new roots reserve fresh IDs; mapping existing customer/library data acquires that customer lock; replace acquires the project exclusive lock; templates-only acquires the template-catalog lock. The transaction rechecks source/plan/target revisions and exact owner token; inserts every remapped document and asset reference; performs transaction-safe replace releases where applicable; and writes one operation/audit result plus exact cleanup intent. Crash/failure before commit leaves no visible record and only recognized staging/prepared orphans for cleanup. After commit the complete new graph is visible at once. Post-commit removal of now-unreferenced old files is an idempotent ledger step and never rolls back the successfully committed replacement.
19. Deletion starts with a read-only, deterministic `DeletionInventory` containing exact target/revision, every owned document/job/export, every media reference, shared reference counts, preserved shared objects, blockers, and an inventory hash. Opening the report mutates nothing. Confirmation requires target ID/revision, inventory hash, a fresh idempotency key, `confirmPermanent: true`, and the displayed target name. Any changed dependency or job revision rejects and returns a fresh report.
20. Confirmed deletion creates a durable hierarchical exclusive scope lock and operation, then force-cancels every nonterminal owned provider/local/human-gate job in the same transaction that removes product records and decrements media references. A customer lock conflicts with every descendant project and customer/family-owned Studio mutation; a project lock conflicts with that project and its owning customer lock while unrelated projects may continue. Every scoped domain write and scheduler enqueue, claim, resume, promotion, running transition, and result commit checks admission in its own transaction. Running workers lose their target/current-state and lock preconditions; late results cannot recreate deleted data. No new project/Studio/export/import work may enter the locked scope. Locks never expire by time and release only through owning-operation recovery after verified cleanup.
21. Physical removal is refcount-aware and resumable. Zero-reference derived/original/export files are unindexed and queued by exact managed record/checksum for idempotent unlink+directory sync. Bytes still referenced outside the deletion scope are preserved and reported as `shared_reference_preserved` without exposing foreign customer identity; the target retains zero link to them. Unknown files or paths outside managed roots are never removed. Startup and operator retry resume incomplete ledgers until every expected DB absence, refcount, managed-file absence/preservation and scope query verifies.
22. A successful `DeletionReport` records counts and stable IDs/checksums by category, canceled job IDs, removed versus shared-preserved media, managed exports removed, verification timestamp, and zero failed checks. It contains no customer text, child-image bytes, phone numbers, paths, secrets, or foreign-owner identifiers. Failed filesystem verification keeps the operation visibly `cleanup_required`; the product data does not reappear and completion is never claimed early.
23. Reusable-character ambiguity is fail-closed. Project deletion always preserves shared library characters and lists them as preserved. Customer deletion necessarily cascades its same-customer projects and Studio history. If the operator asks to keep any older project's pinned character data, the permanent customer deletion is canceled and the non-destructive archive/export path is offered; Hekayati never says a character/photo was permanently deleted while pinned copies remain.
24. Feature participants are additive. Features 008/009 register their exact preview/approval/print documents and PDF/profile assets; the 010 checkpoint round-trips the real 003–009 graph. Slice 010 also proves a synthetic customer-owned 011 participant can register/export/import/delete and that omission fails. Feature 011 must register its real customer-scoped Studio records/jobs/assets and repeat the combined round-trip/customer-delete evidence before its own checkpoint and Phase 10. Prompt-only Studio records with no customer/family owner remain out of customer/project scope. A final completeness test fails if any registered upstream collection or asset reference is omitted.
25. Import/export/deletion endpoints reuse the canonical authority/origin/CSRF boundary, return `Cache-Control: no-store`, require exact scope/revision/hash/idempotency inputs, bound every response, and expose no paths or raw archive bytes except the verified download stream. The Arabic RTL workspace shows pause/quiescence, archive content, migration/conflict plan, destructive replace/delete scope, shared-preserved items, progress/recovery, the no-backup warning, and exact actionable failures with keyboard/focus/44 px/three-width evidence.
26. FR-160's closed action set is exact: `export_pause | export_start | import_upload | import_plan | import_commit | replace_commit | deletion_confirm | deletion_cleanup_retry`. Operation scope is the project for export, the installation for upload before an import operation exists, the import operation for plan/commit/replace, and the typed deletion target/operation for confirmation/cleanup retry. Every action has an action-scoped idempotency key plus canonical request hash persisted atomically with its exact bounded result/state boundary; import upload's hash includes a declared archive checksum and byte count verified by its first accepted stream. Same scope/action/key + same hash returns the stored result without repeating durable work; the same key + a different hash conflicts with zero mutation. Restart/retry creates no duplicate archive, import operation/plan, root graph, reference delta, unlink, or report.

## Slice acceptance scenarios

- **A-010-01 — Quiescent consistent snapshot**: start with queued, blocked, running, waiting-review and terminal project jobs; one-click pause stops new claims, waits/cancels running work explicitly, then a concurrent-mutation fixture proves the exported document/media snapshot is one exact version or fails with no archive. Work remains paused after success.
- **A-010-02 — Complete minimal graph**: export a synthetic completed project containing character photos/originals, sheets, story/scene/prompt/page histories, approvals, preview, RGB/CMYK print evidence and a synthetic customer-owned 011 participant record/asset. Every reachable registered record/asset appears once; another project/customer, global settings/logs/incidents, prompt-only unowned fixture and prior exports appear zero times. Slice 011 repeats this scenario with real Studio history.
- **A-010-03 — Manifest and secret gate**: v2 manifest entries, sizes, hashes, schema versions, ordering and snapshot hash verify; seeded Gemini/Codex/Keychain/bearer/runtime canaries in JSON, binary, entry name and ZIP-boundary fixtures each fail before ready/downloadable state. Clean archive plus DB/log scan yields zero credentials.
- **A-010-04 — Hostile archive matrix**: corrupt/truncated/encrypted/multi-disk, missing/extra/duplicate/checksum-mismatched entries, traversal/absolute/backslash/NUL/Unicode/case collision, symlink/non-regular/executable/nested archive, every exact `ArchivePolicy/v1` boundary and over-boundary zip-bomb/limit, and MIME/schema/reference violations all reject before product writes with one specific bounded reason.
- **A-010-05 — Version policy**: v1 fixture migrates purely to v2 and imports; unknown older, future manifest, missing document migration and future document schema each reject before commit. Source archive checksum remains unchanged.
- **A-010-06 — Disk preflight**: boundary/overflow/false-size fixtures prove insufficient space fails before extraction; a later free-space drop fails before commit and cleans staging/prepared files without visible state.
- **A-010-07 — As-new remap**: import into a populated instance with collisions across every entity/job/asset namespace; every declared ID reference and dependency resolves to the fresh/mapped local graph, every derived/request/idempotency/authorization/provenance hash is recomputed and validates with no stale archive hash, exact compatible bytes deduplicate only by checksum+metadata, and source/unrelated records stay byte-identical. Existing-customer mapping accepts only an exact-revision same-customer attestation, preserves local contact/current-consent authority, and rejects a different-customer shortcut.
- **A-010-08 — Replace**: stale plan, wrong owner/family, unconfirmed scope and changed target revision do nothing. A confirmed plan cancels target work and atomically replaces only its project graph; shared library/Studio/other-project bytes and records remain exact.
- **A-010-09 — Selective modes**: characters-only creates no project/output/job; templates-only contains no customer/private reference and strips a frozen v1 legacy reference into role slots; mode-confused entries reject rather than being silently dropped.
- **A-010-10 — Atomic interruption**: real process kills during validation, staging, prepared-file promotion, SQLite commit boundary and post-commit cleanup recover to either zero imported graph or one complete graph, never a partial visible project; recognized residue is removed or resumed.
- **A-010-11 — Project deletion**: report lists complete project graph/jobs/exports/media and shared library/Studio items as preserved; confirmation cancels jobs, removes project records and zero-ref files, decrements shared refs, and verifies every listed postcondition while unrelated state stays byte-identical.
- **A-010-12 — Customer deletion**: report covers all families/versions/photos/originals/projects/previews/print/exports/jobs plus the synthetic registered 011 participant. Stale confirmation fails; exact confirmation survives running-worker late return and restart, then verifies zero customer-scoped DB/media/export link with honest shared-byte preservation. Slice 011 repeats the deletion with real customer-scoped Studio records.
- **A-010-13 — Failure and recovery**: EACCES/ENOSPC/missing-file/checksum/refcount/DB interruption fixtures keep deletion `cleanup_required`, never claim success or touch unknown/out-of-root files, and converge idempotently after remediation without resurrecting data.
- **A-010-14 — Scoped API and Arabic UI**: wrong customer/family/project/revision/hash/idempotency, raw paths, forged requests and unready downloads fail pre-body/bytes with zero mutation; export/import/delete flows pass Arabic RTL axe/keyboard/focus/44 px/reduced-motion/long-text/no-overflow checks at 390×844, 1440×900 and 1920×1080.
- **A-010-15 — Release evidence**: clean install/check/build/audit/coverage, hostile fixtures, real restart matrix, full 003–009 project+print round trip with the synthetic 011 registration seam, permanent deletion verification, exact replay plus key/hash-collision tests for every FR-160 action, DB/log/archive secret scans and staged repository privacy scans pass with synthetic data only and are recorded in implementation notes. Real Studio integration is repeated in slice 011/Phase 10.

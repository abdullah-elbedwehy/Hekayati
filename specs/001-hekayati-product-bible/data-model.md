# Data Model: Hekayati

**Feature**: `001-hekayati` | **Date**: 2026-07-14
Document-oriented model (C-01, research R2). Every document: `id` (ULID), `createdAt`, `updatedAt`, `schemaVersion`. Collections listed with key fields; free-form maps allowed where noted (NoSQL flexibility). Validation via zod at the repository boundary.

## Conventions

- **Versioned entity**: immutable version documents + a head pointer. `<Entity>Version` holds content; `<Entity>` holds identity + `currentVersionId` + lifecycle status. Nothing edits a version in place; edits append a version.
- **Reference**: by `id` (+ `versionId` where the spec requires version binding). Mentions, approvals, provenance, and job inputs always bind versions.
- **Soft states**: `archived` hides from pickers without breaking references; **permanent delete** is the only destructive operation (FR-005).

## Collections

### customers
`name`, `whatsapp`, `notes`, `consent: null | { granted: bool, date, note }`, `status: active|archived`. `null` is fail-closed "not recorded"; both recorded decisions carry their own date and note (C-13). Archive/restore is non-destructive; permanent deletion is FR-005 only (C-18).
Owns → families. Deletion cascades (FR-005) with pre-report.

### families
`customerId`, `name`, `anchorCharacterId?`, `status: active|archived`. Members are queried from `characters.familyId`; no duplicated member-ID array is stored. A family may be empty, but its first active member MUST have current `relationship.type=main_child`; that creation assigns the anchor only from null in the same transaction. The anchor is immutable until FR-005 deletion, no other active member may have `main_child`, and the anchor's relationship cannot change. A missing or archived anchor blocks later member creation and new Project/Studio selection with `FAMILY_ANCHOR_REQUIRED` or `FAMILY_ANCHOR_ARCHIVED`; it never reinterprets existing relationship versions. Relationship labels are relative to that anchor (C-21). Scoping boundary for project character selection (FR-003).

### characters *(versioned)*
Identity: `familyId`, `status: active|archived`, `currentVersionId`.
`CharacterVersion.profile`: `name`, `nickname`, `relationship { type: main_child|father|mother|brother|sister|grandfather|grandmother|friend|teacher|pet|custom, customLabel? }`, `appearanceDescription`, `ageOrRange`, `gender`, `skinTone`, `hair`, `eyeColor`, `relativeHeight`, `build`, `distinguishingFeatures[]`, `glasses`, `hijab`, `accessories[]`, `interests[]`, `favoriteObjects[]`, `favoriteColor`, `personalityTraits[]`, `speakingStyle`, `notes`, `sourceMode: photo|description|both`, `referencePhotoIds[]`, `traits: map` *(free-form)*. `customLabel` is required only for `custom`; description/both modes require `appearanceDescription`, and photo/both modes require at least one usable reference. Pet status is derived from `relationship.type`; it is not duplicated on identity. `familyId` is immutable in v1. Relationship is version-bound and family-level, while narrative role remains project-only (FR-010/015/017).
Looks are queried from `looks.characterId`; no duplicated look-ID array is stored. Version bump emits the classified change events below; approval superseding and downstream staleness are consumed later by feature 007 (FR-033).

### looks *(versioned)*
Identity: `characterId`, `status: active|archived`, `currentVersionId`. `LookVersion: { name, clothing, appearanceOverrides: map, referencePhotoIds[] }` (e.g., الملابس الأصلية / يومي / بدلة فضاء). Editing rules FR-014: project-only override lives on the project (below), not here. Look names are versioned because they are visible/pinned content; look references resolve through the same consent/privacy-safe `ReferencePhoto.providerAssetId` path as character references.

### referencePhotos *(immutable intake records)*
`customerId`, `familyId`, `owner: { type: character, characterId } | { type: look, characterId, lookId }`, `kind: face|full_body|clothing|other`, `originalAssetId` *(references the private `originalAssets` namespace)*, `workingAssetId`, `thumbnailAssetId`, `providerAssetId?`, `subjectSelection? { x, y, width, height }` *(normalized 0–1 coordinates on the oriented working copy; required for `face`)*, `quality { policyVersion, metrics { widthPx, heightPx, blurScore, exposureScore, shadowFraction, subjectBoxAreaRatio? }, warnings[] { code, source: local_check|operator, metric?, threshold?, details? }, observations { peopleCount?, obstruction?, filterSuspected?, apparentAgeBand?, hair?, clothing? } }`, `usableAsFaceReference: bool`, `supersedesPhotoId?`.

The exact upload remains local in the separate original namespace. `workingAssetId` is always a newly derived, orientation-corrected, metadata-stripped `reference_photo` asset, so it can never alias the original record. `providerAssetId` is the only field later provider code may accept: for `face`, it is always a newly derived sanitized subject crop; for a suitable `full_body`, `clothing`, or `other` reference, it may equal the privacy-clean working asset. Face intake cannot commit until a subject rectangle exists, and multi-person input requires an explicit intended-person selection (FR-021/024/134). Character-owned records must occur in the pinned `CharacterVersion.referencePhotoIds`; look-owned records must occur in the pinned `LookVersion.referencePhotoIds`, whose `characterId` must match. Originals are not `Asset` IDs and are structurally ineligible for provider payloads.

Photo intake uses a private pre-commit reservation, not a persisted product collection. A cryptographically random runtime-only token identifies generated `.hekayati-tmp-*` files and validated safe metadata; it is carried only in CSRF-protected request bodies/headers and is absent from URLs, logs, exports, and final records. The reservation targets an existing character/look or a preallocated new-character draft. It computes the family-local duplicate candidates after content hashing and returns only a derived thumbnail plus safe findings. Commit requires the operator's duplicate decision and every required face rectangle. For a new photo-only character, the character identity, first usable `CharacterVersion`, `ReferencePhoto`, original/derived asset records, head, and outbox events commit together. For an existing character/look, the same transaction appends the owning version and advances its expected head. Cancel/expiry compensates prepared files; restart loses the runtime token and startup GC removes only the recognized unindexed reservation files (FR-019/025).

### characterSheets
`characterId`, `characterVersionId`, `appearance: { type: base, lookId: null, lookVersionId: null } | { type: shared_look, lookId, lookVersionId }`, `views { face, front, threeQuarter, fullBody, mainOutfit } → assetIds`, `referenceLineage { source: description_only|photo_derived, referencePhotoIds[] }`, `status: generating|ready|revision_needed|approved_superseded`, `pdfAssetId`, provenance. `referenceLineage` is derived from the pinned input versions and generation provenance, never accepted from a caller: any transitive photo input makes the sheet `photo_derived`; only zero-photo lineage is `description_only`. Base appearance never invents a look ID; shared-look appearance requires exact owned look/version IDs. Approval target (FR-030–033) and later FR-004 consent decision input.

### projects *(versioned configuration)*
Schema v2 identity: `customerId`, `familyId`, `revision`, `status` (see state-machines.md), `priority`, `paused: bool`, `currentVersionId`, `bookVersion` (monotonic; bumps on any customer-visible change — drives FR-086), `compositionProfileId`, `currentCoverCompositionVersionId?`, `currentPreviewOutputId?`, `currentPreviewCycleId?`, `currentContentApprovalId?`, `printerProfileId?`. Customer/family ownership is immutable. Every project/head/status mutation compare-and-swaps `revision`; cover/preview/cycle/authorization heads are expected-revision pointers, and advancing either never mutates the referenced version. Preview commit advances `currentPreviewOutputId/currentPreviewCycleId` only. `currentContentApprovalId` advances only when that exact cycle's gate succeeds by approval; it remains on the prior approved cycle when a new unapproved or watermark-only preview cycle is created. It is revoked only by a matrix customer-visible ✖ row or an explicit changes-requested outcome on the same customerContentHash. The v1→v2 repository migration preserves identity/status/currentVersionId/bookVersion/printerProfileId/timestamps exactly, initializes `revision: 0`, pins the built-in A4 portrait composition profile, and sets all new downstream heads to null. Project and PageLayoutHead migrations run in one startup transaction; a partial/duplicate/invalid migration rolls back and restart repeats safely.

`ProjectVersion.storyConfig`: `title`, `mainChildId`, `participants[] { characterId, characterVersionId, narrativeRole, appearanceSelection }`, `occasion`, `dedicationText`, `storyType: connected_adventure|related_situations|saved_template|fully_custom`, `templateId? + templateVersionId?`, `pageCount: 16|24`, `tone: light_funny|adventurous|warm_family|magical|educational_non_preachy|custom`, `customTone?`, `illustrationStyleId: modern_cartoon|colorful_2d|soft_watercolor`, `hiddenGoal { goal, customGoal?, presentation: indirect|acknowledged_ending } | null`, `clothingNotes`, `customNotes`, `audienceAgeBand: age_3_5|age_6_8|age_9_12`, `readingLevel: early|developing|independent`, `sceneComplexity: low|medium|high`, `narrationDialogueBalance { suggestedNarrationPercent, selectedNarrationPercent, operatorEdited, formulaVersion }`, `customStory? { premise, beginningBeat, middleBeat, endingBeat, contentBoundaries[] }`, and editable `endingPages { farewellText, brandLine }`.

Every participant is same-family and pins an immutable character version. `mainChildId` is one selected non-pet participant; the family's assign-once anchor must independently be active (C-21). `appearanceSelection` is the strict union `base | { sharedLook: { lookId, lookVersionId } } | { projectOverride: { overrideId, overrideVersionId } }`. The selected template version is pinned; a `saved_template` requires it, while `fully_custom` uses C-23. Configuration edits append under expected-head compare-and-swap; prior versions remain readable.

The canonical customer-visible page map is a deterministic projection of the pinned `ProjectVersion`: page 1 title, page 2 dedication, pages 3–14 or 3–22 story, then farewell and brand ending pages. Covers and printer-only blanks are absent. A page-count preflight returns an explicit add/merge/remove mapping plus a content hash; committing it requires that hash, the expected project head, and operator confirmation, then appends one project version. It never edits or drops authored scenes as a side effect (FR-055–058, C-05).

### projectCharacterOverrides *(versioned)*
Identity: `projectId`, `characterId`, `currentVersionId`, `status: active|archived`. `ProjectCharacterOverrideVersion: { baseCharacterVersionId, baseLookVersionId?, clothing, appearanceOverrides: map }`. It may reference only the project's pinned character and an owned pinned look; it introduces no new photo intake path. Create/edit appends the override version, advances its expected head, updates the project participant's pinned `appearanceSelection`, and emits one IM-04 `changeEvent` atomically. Shared `CharacterVersion` and `LookVersion` records remain byte-identical (FR-014(a)).

### storyTemplates *(versioned)*
Identity: `seedKey?`, `status: active|archived|disabled`, `currentVersionId`. `TemplateVersion: { name, premise, structure[], environments[], roleSlots[] { slot, requiredRelationship?, narrativeRole }, variables[] { key, type, required, default? }, possibleHiddenGoals[], sceneGuidance[], ageAdaptationRules[], contentBoundaries[], endingPatterns[] }`. Template edits append under expected-head compare-and-swap; projects pin `templateVersionId` forever (FR-050–052). `disabled` stays visible in management but is unavailable to new projects; `archived` is hidden from routine lists; either remains readable by pinned projects and is reversible. Seven stable-key seed templates install idempotently at first run; restart never overwrites an edited or disabled/archived seed (FR-053).

### stories *(versioned)*
Identity: `projectId`, `status: draft|complete`, `currentVersionId`. `StoryVersion: { previousVersionId?, source: manual|generated, planJson? (validated StoryPlan schema), sceneVersionIds[], pageCountChange? { from, to, planHash, operations[] }, completedAt? }`. Manual authoring in feature 004 and generated content in feature 007 use the same immutable lineage. `complete` requires one valid scene version for every story-page slot; provider generation is not required. Regenerating or editing appends a version; scenes/pages pin `storyVersionId`. A page-count confirmation appends a structure revision: source scene versions remain immutable, while added/merged targets are explicit incomplete drafts until authored.

### scenes *(versioned)*
Identity: `projectId`, `storyPageIndex`, `currentVersionId`. `SceneVersion: { previousVersionId?, sourceSceneVersionIds[], needsAuthoring, purpose, description, documentSegments[], environment, timeOfDay, composition, cameraFraming, narrativeText, dialogue[] { speakerCharacterId, text }, twoImageMoment: bool }`. `storyPageIndex` is 1–12 or 1–20 and maps to customer-visible page 3 onward; it is not a printer page number. `sourceSceneVersionIds` is empty for ordinary manual authoring and records only explicit page-count retain/merge provenance; it never grants mutable aliases.

`documentSegments` is an ordered strict union: `{ type: text, text } | { type: mention, characterId, props { action, emotion, position?, framing?, lookId?, heldObject?, gazeTarget?, speaks, dialogue? } } | { type: group, groupKey: hero|friends|family, props? } | { type: unresolved, text }`. Mention documents store no authoritative display name and no character version: the editor renders the character's current display name while compile resolves the project's pinned `characterVersionId`. A partial delete or paste can only produce a complete token or an explicit unresolved segment; unresolved segments block compile. Built-in group expansion follows C-24. Dialogue speaker IDs must occur in the confirmed scene participant set (FR-035–041).

### pages *(versioned lineage per FR-059)*
`projectId`, `pageNumber`, `kind: title|dedication|story|ending1|ending2`,
`locked: bool`, `reviewStatus: unreviewed|flagged|approved`, `staleReason?` (from invalidation),
`currentIllustrationVersionId?`. Page layout identity/head is downstream in `pageLayoutHeads`, so deriving the first layout from an exact frozen creative snapshot does not mutate a locked Page record.
`IllustrationVersion`: `assetId`, `promptCompiled`, `negativeConstraints[]`, `inputSnapshot { sceneVersionId, characterVersionIds[], lookVersionIds[], styleId, settingsHash }`, `provenance`, `supersededBy?`.

### compositionProfiles, pageLayoutHeads, and layoutVersions *(feature 008)*
`CompositionProfile` is versioned customer-composition geometry, distinct from a printer profile: `id`, `version`, `trimWidthMm`, `trimHeightMm`, `dimensionToleranceMm`, normalized safe content region, approved placement regions, typography scale, and hash. V1 is the explicit A4 portrait composition canvas with a declared 0.5 mm dimension tolerance. Layout and approval bind this profile. A printer profile is composition-compatible only when it remains portrait, both trim dimensions match within that pinned tolerance without scaling, and the composition safe-content rectangle is wholly inside the printer safe rectangle after the identity trim mapping; bleed/DPI/color/ICC/crop/spine/blank-page values do not enter this predicate. Anything else hard-blocks and requires an explicit composition-profile migration that creates new layouts/cover composition and a new approval cycle rather than masquerading as IM-14.

`PageLayoutHead`: `pageId` (unique), `currentLayoutVersionId`, `revision`, `updatedAt`. Initial story derivation may create this downstream head from an exact approved-review/locked creative snapshot because the Page and creative heads remain byte-identical; special-page derivation instead uses exact ProjectVersion/policy/selection/template inputs and requires no nonexistent PageReview. Replacing/recalculating an existing layout is customer-visible and requires the Page to be explicitly unlocked.

`LayoutVersion`: immutable `pageId`, `previousVersionId?`, `inputSnapshot { compositionProfileId, projectVersionId, pageObservationRevision, pageContentHash, pageReviewId?, reviewHash?, textVersionId?, illustrationVersionId?, templateVersion, compositionSourcePolicyVersion?, selectionSource: automatic_v1|operator|not_applicable, compositionInputHash, textSources[] { role, entityId, versionId, contentHash }, sourceAssets[] { role, assetId, checksum }, typographySettingsHash, fontManifestHash }`, `requestedPlacement: auto|top|bottom|right|left`, `resolvedPlacement`, `resolvedRegion`, `readabilityAid: none|gradient|panel`, `fontSizePt`, `overflow: bool`, `warnings[]`, `acceptance: ready|needs_operator`, `bubbles[] { speakerCharacterId, text, region, pointerAnchor? }`, `measurementHash`, `layoutPolicyVersion`, `rendererVersion`, `workRequestId?`, `jobId`, and canonical `layoutHash`. Story pages require an exact approved PageReview ID/hash and `selectionSource: not_applicable`; special pages forbid review fields and require a composition-source policy/selection. Bounded ordered text sources pin every exact ProjectVersion/PageTextVersion/CharacterVersion/brand-template source, including the child-name version rendered on title/brand pages. Source assets pin story illustration media and every selected sheet/hero/story/decorative/bundled identity asset. `pageObservationRevision` is a job-CAS fence only: lock/review/stale metadata changes may force a running job to retry but do not enter `pageContentHash` or later authorization. `pageContentHash` covers only customer-visible page text/art/assets; `compositionInputHash` adds text refs, source policy/selection, assets, and template input; `layoutHash` binds the entire immutable layout. A missing/changed input or checksum rejects commit; an indeterminate speaker uses a labeled non-pointing fallback and warning rather than a guessed pointer.
Commit precondition: job's `inputSnapshot` must match current lineage else reject (FR-065).

### coverCompositions *(feature 008; versioned customer-visible content)*
Identity: `projectId`, `currentVersionId`, `revision`. `CoverCompositionVersion`: `compositionProfileId`, `previousVersionId?`, `projectVersionId`, `compositionSourcePolicyVersion`, `selectionSource: automatic_v1|operator`, bounded ordered `textSources[] { role, entityId, versionId, contentHash }`, selected approved hero/sheet/story asset IDs and checksums, `front { title, childDisplayName, environmentLine?, artworkAssetId }`, `back { synopsis?, brandLine, artworkAssetId? }`, local brand-template/font hashes, normalized front/back regions, closed `warnings[] { code, severity: advisory|blocking }`, `acceptance: ready|needs_operator`, and `compositionHash`. The automatic v1 text policy pins title, child display name, and ending-page brand line from the exact ProjectVersion/main-child CharacterVersion and leaves environmentLine/synopsis absent; the source refs, policy/selection, warnings, readiness, and resolved optional-value representation are covered by `compositionHash`, and only the explicit revision-checked cover edit may add them. Any blocking warning requires `needs_operator`; only `ready` can enter `pdf_pending`. It is a customer-view composition only: no spine, bleed, crop marks, ICC, or printer spread. Preview renders watermarked front/back proof pages; feature 009 maps the approved version into actual back+spine+front geometry. Editing it emits IM-12.

### previewWorkflows *(feature 008; durable automatic continuation)*
One mutable workflow per project: `projectId`, `revision`, `state: layout_pending|operator_action_required|pdf_pending|rendering|validating|ready|failed`, `inputSnapshotHash`, `layoutJobIds[]`, `previewJobId?`, `blockingReasons[]`, `currentPreviewOutputId?`, `updatedAt`. Completing the 007 internal-review gate idempotently creates/refreshes this workflow and strict local layout jobs. When every required layout and cover composition is ready, one transaction persists `pdf_pending` and materializes the preview job. It survives restart and never implies a send/approval action.

### previewOutputs *(feature 008; content immutable, status projected)*
`projectId`, `assetId`, `jobId`, immutable `approvalCycleId`, immutable `approvalGateJobId`, `bookVersion`, `projectVersionId`, `compositionProfileId`, `coverCompositionVersionId`, `customerContentHash`, `orderedInteriorPages[] { pageId, pageNumber, pageObservationRevision, pageContentHash, pageReviewId?, reviewHash?, layoutVersionId, textVersionId?, illustrationVersionId?, compositionSourcePolicyVersion?, selectionSource, compositionInputHash, textSources[] { role, entityId, versionId, contentHash }, sourceAssets[] { role, assetId, checksum } }`, `approvalBundleHash`, `pageMapHash`, `previewSnapshotHash`, `watermarkSettingsHash`, `previewDerivativePolicyHash`, `typographySettingsHash`, `fontManifestHash`, `rendererVersion`, `validationReport { interiorPageCount, totalProofPageCount, orderedPageMapPassed, parsePassed, pageBoxes[] { pageIndex, mediaBoxMm, trimBoxMm?, portrait, tolerancePassed }, compositionDimensionsPassed, embeddedFontIds[], toUnicodePassed, watermarkPages[], footerPages[], imagePpi[], bytes, sizeBudgetPassed, externalRequestCount, prohibitedPdfFeatureCount }`, revisioned projection `revision`, `status: ready|stale`, `staleReasons[]`, `invalidatedByEventIds[]`, `createdAt`. `customerContentHash` covers only composition/cover/order/pageContent/layout/text/source/full-resolution customer-visible fields, never preview/cycle/gate/watermark/derivative/status/attention/operational revisions. The cycle is created at preview commit in `ready_to_send` revision 0 and the gate request targets this exact PreviewOutput ID/version. Every page entry repeats exact review evidence where applicable, policy/selection, composition-input hash, text refs, and assets from its LayoutVersion. `approvalBundleHash` additionally binds the exact PreviewOutput, review evidence, watermark and derivative policy. Every page's MediaBox and applicable TrimBox must match the pinned CompositionProfile dimensions within its declared tolerance and remain portrait. The PDF contains a customer-view front-cover proof, exactly 16/24 interior pages, and a back-cover proof; cover proofs do not change interior numbering and contain no printer geometry. Content/snapshot fields never change; invalidation may compare-and-swap only `revision/status/staleReasons/invalidatedByEventIds`. The project head selects the current ready candidate, while older outputs remain auditable until feature-010 deletion.

### jobs *(scheduler-owned — full semantics in contracts/job-scheduler-contract.md)*
Strict version-1 document with `type`, `projectId?`/standalone scope, `dependsOn[]`, `state` + bounded reason, `priority: 1..5`, `revision`, `intentId`, `idempotencyKey` (unique), independent canonical `requestHash`, immutable `target: { provider, modelId, operation, settingsHash } | null`, canonical persisted request (`TextRequest`/`StructuredRequest`/`ImageRequestDraft`, strict `{ kind: local, payloadHash }`, or human-gate descriptor), `inputSnapshot { entity IDs + version IDs }`, `lease? { workerId, bootId, claimToken, expiresAtMono }`, attempt/retry counters, progress/stall projection, normalized privacy-safe failure/structural diagnostics, provenance, `resultRefs[]`, and optional predecessor/successor job links. Provider requests require a non-null exact target; local and human-gate requests require `target: null`. A registered local definition resolves its bounded descriptor from an owning strict repository by `payloadHash`; request documents never contain runtime bytes, paths, secrets, arbitrary payloads, or provider fields. Runtime bytes/paths/`ResolvedImageRequest`, originals, secrets, raw provider output, rejected values, command output, and arbitrary provider payloads are invalid. Append-only `jobEvents` retain bounded transition/attempt/progress/failure/decision/rejection/recovery history. Provider quota incidents and the global storage pause are persisted; audit events record only explicit wait/continue decisions and successor links.

### originalAssets *(private intake namespace; never a provider input type)*
`sha256` (unique within this namespace), `sourceMime`, `extension`, `bytes`, `refCount`, `createdAt`. Exact upload bytes live at `originals/<sha256[0:2]>/<sha256>.<ext>` with the same atomic-write, checksum, private-permission, integrity-scan, and reserved-name-only GC rules as R4. This namespace deliberately has no provider provenance or provider-facing ID conversion. It exists only for FR-021 local retention, export, integrity, and FR-005 deletion.

### assets
`sha256` (unique), `mime`, `bytes`, `width/height/dpi?`, `role: reference_photo|sheet_view|illustration|pdf_preview|pdf_interior|pdf_cover|thumbnail|import_staging`, `origin: upload|generated|derived`, `provenance { provider, model, at, jobId, inputVersionRefs, promptVersion, referencedAssetIds[], attempt, settingsSnapshot }` (FR-094), `exifStripped: bool`, `refCount`.

`settingsSnapshot` is a strict, versioned, secret-free generation-settings record: a required SHA-256 `settingsHash` plus only explicitly modeled controls such as quality mode, style ID, reference budget, economy-tier state, and output dimensions. Provider response payloads, arbitrary nested values, credentials, prompt/image bytes, and runtime tokens are invalid. Generated assets require provenance. Every upload-derived `reference_photo` and `thumbnail` asset requires `exifStripped: true`; for this flag, “EXIF” means verification that GPS, EXIF, IPTC, XMP, device, and other non-essential metadata are absent. Because `sha256` is globally unique while role/origin/provenance are singular, a same-byte put may increment `refCount` only when canonical metadata is identical; conflicting metadata fails explicitly instead of discarding traceability. Exact uploads never collide with this rule because they live in the private original namespace, while every working/provider asset is a derived copy.
File at `assets/<sha256[0:2]>/<sha256>.<ext>`; write path per R4.

The data root itself is application-owned only after a valid `.hekayati-data-root.json` marker is created in an empty root. Reusing a non-empty unmarked root is invalid; managed child paths cannot be symlinks. Orphan collection recognizes only Hekayati temporary names and canonical content-addressed filenames.

### approvals
Character approval keeps the feature-007 sheet-bound contract. A book approval is one version-checked preview cycle created atomically with its ready PreviewOutput: `kind: book`, `projectId`, immutable `previewOutputId`, immutable `approvalGateJobId`, `targetBookVersion`, `customerContentHash`, `approvalBundleHash`, `pageMapHash`, `previewSnapshotHash`, `coverCompositionVersionId`, `watermarkSettingsHash`, mutable `state: ready_to_send|preview_sent|approved|changes_requested|invalidated`, initial `revision: 0`, `notes`, bounded strict `affectedScopes[]: { kind: page, pageId } | { kind: cover, side: front|back|both }`, `recordedAt`, `invalidatedBy? { eventId, matrixRow, at }`, `attentionReasons[]`. Page scopes must belong to the bound PreviewOutput and cover scopes to its exact CoverCompositionVersion; duplicates/foreign IDs reject. The Project `currentPreviewCycleId`, PreviewOutput `approvalCycleId`, cycle, and gate must cross-reference exactly; the gate request targets that output. Approval also advances `currentContentApprovalId` to the cycle in the action transaction. A later ready/unapproved cycle never replaces that authorization head. Every action compare-and-swaps the non-null cycle and gate revisions. Only state/revision/notes/scopes/attention may change; every transition appends audit evidence. `preview_sent` and `approved` require the exact current ready output and matching customerContentHash. `attentionReasons` implements matrix ⚠ without changing approval state (for example IM-19); it cannot be used to approve a stale output (FR-085–087, C-26).

`BookApprovalAction` is the append-only restart-safe idempotency ledger: `cycleId`, unique scoped `idempotencyKey`, `canonicalRequestHash` (including normalized notes and affectedScopes), `action: preview_sent|approved|changes_requested`, exact input `projectRevision`, `previewOutputRevision`, `approvalRevision`, `gateRevision`, `expectedContentApprovalId?`, `expectedContentApprovalRevision?`, immutable target IDs/hashes, `result { projectRevision, previewOutputRevision, approvalRevision, gateRevision, currentContentApprovalId?, projectStatus, approvalState, gateState }`, and `recordedAt`. Inserting this ledger row, compare-and-swapping project/output/cycle/gate/current-content-approval state, and appending audit evidence occur in one transaction. Same scoped key + same request hash returns the stored result; same key + different hash is a collision; a different key with stale revisions changes nothing. A same-customerContentHash changes-requested outcome invalidates the prior authorized cycle and clears the authorization head in this transaction.

`ApprovedBookSnapshot` is a strict transient reader result, never a mutable collection: `projectId`, `projectVersionId`, `bookVersion`, `approvalCycleId`, `approvalGateJobId`, `previewOutputId`, `approvalBundleHash`, `customerContentHash`, `compositionProfile { id, version, hash }`, `cover { versionId, compositionHash, compositionSourcePolicyVersion, selectionSource, textSources[] { role, entityId, versionId, contentHash }, sourceAssets[] { role, assetId, checksum } }`, ordered `pages[] { pageId, pageNumber, pageContentHash, pageReviewId?, reviewHash?, layoutVersionId, layoutHash, textVersionId?, textHash?, illustrationVersionId?, compositionSourcePolicyVersion?, selectionSource, compositionInputHash, textSources[] { role, entityId, versionId, contentHash }, sourceAssets[] { role, assetId, checksum } }`, stable `contentAuthorizationHash`, and non-hashed `observationRevisions { project, pages[], approval, gate, previewOutput }`. `customerContentHash` compares content across preview cycles. `contentAuthorizationHash` covers that hash plus immutable approved preview/cycle/gate outcome/review evidence, but explicitly excludes mutable current-preview heads, watermark setting, PreviewOutput status/revision, approval attention/revision, and operational page/project revisions. Thus IM-19 or a newer unapproved preview changes observations without changing the prior authorization. A customer-visible ✖ row invalidates the cycle/guard. IM-20 blocks only while a referenced content asset/output required by print fails exact checksum/integrity; byte-identical repair/reverification can restore the guard without pretending content changed or requiring new approval. The result contains no bytes, filesystem paths, mutable “latest” pointers, preview derivatives, or printer settings. Feature 009 pins the stable hash/field set in every print job and re-runs the reader before dispatch and commit; only a changed hash or blocking guard predicate rejects.

### portability participant registry *(code-owned, not a collection)*

`PortabilityParticipant` is the closed typed ownership boundary for every project/customer-bearing collection, scheduler lineage, managed export, and asset/original role. Each entry declares its schema version and migrations, dependencies, project/customer selectors, asset/original references, allowed export/import modes, field-aware ID rewrite, derived-field/hash rebase, deletion order, and post-delete verifier. The frozen registry is topologically sorted and hashed. Startup/tests fail on a duplicate/cycle or when the explicit known-collection/asset-role/job-lineage catalog contains an unregistered owned type. Slice 010 registers the real 003–009 graph and proves the 011 registration seam with a synthetic participant; slice 011 owns its real Studio participant and repeats the combined evidence before its checkpoint.

### portabilityScopeLocks *(durable operation admission)*

`scope { kind: customer|project|template_catalog, id, customerId?, projectId? }`, `operationId`, `mode: export_snapshot|import_commit|replace_import|permanent_delete`, `phase: draining|snapshot|exclusive|releasing`, `revision`, captured-attempt ledger root/count, `acquiredAt`, `updatedAt`. A customer lock conflicts with that customer and every descendant project; a project lock conflicts with the same project and its owning customer lock; the template-catalog lock serializes template-only commits. Conflict detection, lock insertion, target-revision checks, project pause/job admission, and capture of exact already-claimed/running `{jobId, attempt}` pairs are one SQLite transaction. Only those captured attempts may finish while `draining`; none may commit after transition to `snapshot|exclusive`. Every scope-bearing domain mutation, scheduler enqueue/claim/resume/promotion/running transition/result commit, Studio action, export/import commit, and deletion confirmation checks the same admission service in its mutation transaction. Locks have no time-based lease: startup resumes or fails the owning durable operation closed, and only verified staging/cleanup releases the lock.

### portabilitySnapshots and internal snapshot entries

`PortabilitySnapshot`: `operationId`, exact project/customer/family IDs and revisions, participant-registry hash, `state: freezing|frozen|staging|staged|released|failed`, document/media counts and bytes/hashes, `snapshotHash?`, next ordinal, `failureCode?`, timestamps. The operation acquires its project lock and reaches quiescence, then one short immediate SQLite transaction rechecks the lock/project/jobs, selects the registered closure, and writes the exact schema-validated canonical document bytes plus immutable media/original metadata and checksums into private `portability_snapshot_entries` rows. Those internal rows are keyed by `(snapshotId, ordinal)`, bounded to one document or media item each, carry inclusion reason and generated archive entry name, and contain no local filesystem path. Idempotent `PortabilityMediaHold` rows retain each exact asset/original once for the snapshot. The transaction commits before asynchronous filesystem staging begins. Staging reads only these frozen rows, re-reads held content-addressed bytes by managed ID, and authoritatively verifies streamed byte count/checksum/metadata; it never rereads live documents. Drift fails with no candidate archive. After all bytes are privately staged and verified, one transaction marks `staged`, releases holds with durable zero-ref cleanup intent, and releases the project lock; final packaging and both secret scans consume only staged bytes. Snapshot rows/holds are terminally cleaned only through their owning operation.

### exportOperations and managedExports

`ExportOperation`: owner project/customer/family IDs, revision, exact request hash + idempotency key, project revision, state `waiting_pause|waiting_quiescence|acquiring_lock|freezing_snapshot|staging|packaging|secret_scanning|ready|failed|stale`, snapshot/manifest hashes, bounded counts/bytes, `archiveKey?`, finalized archive checksum, bounded failure/cleanup state, timestamps. `ManagedExport`: export/operation/owner IDs, `archiveKey` *(app-generated managed relative key, never an operator path)*, manifest version, snapshot/manifest/archive checksums, bytes, secret-scan result, createdAt. Same scoped idempotency key + same request hash returns the stored operation; the same key + a different hash conflicts. Failed candidates never replace a prior ready export. Project export excludes prior managed exports; customer permanent deletion inventories and removes its managed exports.

The strict current manifest identity is the pair `format: "HekayatiArchive"` + `manifestVersion: 2`, written as `HekayatiArchive/v2`. The one frozen legacy fixture is `format: "HekayatiArchive"` + `schemaVersion: 1`; its pure staging-only migration produces v2. Both manifest and generated archive entry names are portability identifiers, not local paths.

### importOperations, importPlans, and bounded portability ledgers

`ImportOperation`: opaque managed upload reservation key, revision, exact source archive hash/bytes, manifest/normalized-manifest hashes, `actionRefs { uploadActionId, latestPlanActionId?, commitActionId? }`, mode `as_new_project|replace_existing|characters_only|templates_only`, state `uploaded|validating|plan_ready|committing|imported|rolled_back|failed|cleanup_required`, `planId?`, bounded result-root IDs/counts, failure/cleanup state, timestamps. `ImportPlan` is immutable and pins source/snapshot/manifest hashes, mode, target owner/revisions, customer/family resolution and attestation hash, conflict choices, disk/migration/sanitization summaries, and root hashes/counts for the complete ID-map, rebase, write, release, prepared-media, and authorization ledgers plus a final `confirmationHash`. Large sets never live as unbounded arrays in an operation document.

`PortabilityLedgerPage` is immutable: `operationId`, `ledgerKind`, `pageIndex`, at most 256 strict entries, `pageHash`, `createdAt`. Root hashes cover ordered page hashes and counts. Ledger kinds include snapshot indexes, import ID maps/conflicts/writes/releases/rebases/prepared media, deletion inventory items/unlinks/shared preservation/verification, and report detail. APIs expose bounded pages only. IDs are allocated once at plan creation; participants rewrite only declared fields and recompute every ID/request-derived hash before the complete graph is revalidated.

`PortabilityAction`: `id`, `operationScope { kind: installation|project|import_operation|deletion_target|deletion_operation, id }`, closed `action: export_pause|export_start|import_upload|import_plan|import_commit|replace_commit|deletion_confirm|deletion_cleanup_retry`, `idempotencyKey`, canonical request hash, exact input revisions/hashes, bounded result or result hash, and recordedAt; unique on `(operationScope.kind, operationScope.id, action, idempotencyKey)`. Export uses its project scope; pre-operation upload uses the stable local installation ID and hashes the declared archive checksum/byte count that the accepted stream verifies; planning/commit/replace use the import operation; deletion uses the typed target or cleanup operation. Each action is inserted in the same transaction as its durable result/state boundary. Same scope/action/key + same hash returns it; the same key + a different hash conflicts without rerunning durable work or duplicating an archive, operation, plan, graph, reference delta, unlink, or report (FR-160).

### deletionInventories, deletionOperations, and deletionReports

`DeletionInventory` is immutable and short-lived: target `customer|project` ID/revision/display-name hash, exact job/participant/media/export counts, preserved-shared counts, blockers, ledger root hashes, inventory hash, and createdAt. Opening it mutates nothing. `DeletionOperation` stores target kind/ID hash, inventory/request hashes, scoped idempotency key, state `committing|unlinking|verifying|verified|cleanup_required`, canceled-job and deleted-count summaries, unlink/shared/verification ledger roots, and timestamps. `DeletionReport` is the immutable verified projection with bounded counts, ledger roots, verification timestamp, and zero failed checks. These evidence records contain no customer text, phone number, bytes, secret, path, or foreign-owner identifier and survive removal of the target graph. Same key/hash replays the stored result; key/hash collision fails.

Confirmation atomically acquires the scope lock, rechecks a fresh inventory, force-cancels every owned nonterminal job/gate, removes documents in reverse participant order, applies asset/original/export reference deltas, and persists exact unlink work. Physical managed-file removal and parent-directory sync resume from the ledger after commit. Positive out-of-scope refcounts preserve shared bytes anonymously. Only complete DB/file/refcount/scope verification advances `verified` and releases the lock; failures remain `cleanup_required` without resurrecting product data.

## Versioning & invalidation hooks

- Any write that bumps a version emits the `changeEvents` above. Character fields classify as: `rename` = `name`, `nickname` (IM-05); `permanent_appearance` = `appearanceDescription`, `sourceMode`, `ageOrRange`, `gender`, `skinTone`, `hair`, `eyeColor`, `relativeHeight`, `build`, `distinguishingFeatures`, `glasses`, `hijab`, `accessories`, `referencePhotoIds`, plus unclassified `notes`/`traits` because they may contain appearance guidance (IM-01); `non_visual_profile` = `ageOrRange`, `gender`, `relationship`, `interests`, `favoriteObjects`, `favoriteColor`, `personalityTraits`, `speakingStyle`, plus `notes`/`traits` (IM-02); any `LookVersion` name/clothing/override/reference change = `shared_look` (IM-03); project inline/look selection = `project_look_override` (IM-04). One edit may emit multiple events; the later engine deduplicates consequences rather than dropping a row. Slice 003 persists its character/look/visibility events; slice 004 persists project-override events; slice 007 consumes both when the shared invalidation engine exists. Archive/restore emits IM-21 only and never changes pinned versions. Locked pages receive flags only (FR-064).
- `bookVersion` increments on: page text/illustration/layout change, page order/count change, dedication/title/cover change — the customer-visible set of FR-086. Preview rendering, send/approval actions, watermark-only changes, and internal records do not bump it. A single logical change/event bumps the affected project once even when several downstream artifacts are invalidated.
